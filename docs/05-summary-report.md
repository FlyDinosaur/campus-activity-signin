## 1. 本组项目组织情况

本组以“校园活动报名与签到管理系统”为统一项目背景，按照项目管理实验要求，将工作重点放在范围计划、进度计划、配置管理和 GitHub 协作过程留痕上。吴宇轩负责创建仓库、建立 `main` 与 `dev` 分支、初始化目录结构，并在最终阶段将全组成果合并到 `main` 分支。

在组织方式上，本组采用个人分支协作模式。每名成员均有独立分支，并且每名成员都参与至少一次合并操作，从而保证提交记录能够体现真实协作过程。

## 2. WBS 分解思路
本组 WBS 分解坚持交付物导向原则，以“最终需要提交和检查的成果”为主线。一级交付物包括项目管理成果、范围计划成果、进度计划成果、配置管理成果、示例代码成果和总结报告成果；二级交付物进一步划分为启动成果、WBS 分解成果、活动识别成果、配置项识别成果等；三级交付物细化为可以被成员独立编辑、提交和检查的最小成果单元。
在分解过程中，本组避免将“开会、讨论、修改、沟通”等过程动作作为 WBS 主体，而是将其转化为可交付的文档、表格、记录或代码文件。这样既符合项目管理实验要求，也方便通过 GitHub 提交记录核查成员贡献。

## 3. 进度安排依据
本组进度安排以 WBS 交付物为基础，将每个主要交付物转化为关键活动，并为活动设置⼯期、前置关系和责任⼈。项⽬启动说明和仓库结构是所有后续活动的基础，因此安排在最前；WBS ⽂档完成后，才能进⼀步提取关键活动并编制进度计划；配置管理⽅案与进度计划可以并⾏推进，但冲突解决必须在成员分⽀合并过程中完成。
从责任分配看，吴宇轩负责基础结构和最终集成，李奕轩负责 WBS，赵新宇负责配置管理，伍湘彬负责进度计划和冲突解决验证。该安排能够保证每项成果均有主要责任⼈，同时通过共同编辑 `docs`⽂件体现全员参与。

本组配置管理过程以 GitHub 仓库为核⼼，⾸先建⽴ `main` 和 `dev` 两个公共分⽀，再由四
名成员分别创建个⼈ `feature` 分⽀。每名成员均在个⼈分⽀中编辑 `docs` ⽂件夹下的全部⽂
档，并提交到远程仓库。
在合并过程中，本组采⽤ `--no-ff` ⽅式保留分⽀合并记录。李奕轩负责合并吴宇轩基础分⽀，赵

突，吴宇轩负责最终将 `dev` 合并到 `main`。这种流程能够保证每个⼈都有分⽀、有提交、有合
并记录。

## 6. 经验与问题
通过本次实验，⼩组成员认识到 GitHub 协作不仅需要提交⽂件，还需要保证提交⼈名称、分⽀来源、合并顺序和提交信息规范。如果只关注最终⽂件，很难体现真实协作过程；如果每个⼈都在个⼈分⽀中编辑⽂档并参与合并，则可以更清楚地展示个⼈贡献。
