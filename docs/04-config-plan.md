## 2. 仓库结构
本项目仓库采用统一目录结构进行配置管理：

- `README.md`：项目基本说明、成员角色、分支约定。
- `docs/`：保存本实验最重要的项目管理文档，是主要检查对象。
- `src/`：保存象征性 Java 示例代码，用于体现系统结构。
- `test/`：保存简单测试文件。
- `assets/`：保存资源文件或占位文件。

仓库初始化后，所有成员不得随意修改顶层目录结构。如确需新增目录，需要先在 `dev` 分支中说明原因，再由成员分支提交。
`main` 分支只保存最终稳定成果，普通成员不得直接向 `main` 提交。所有修改应先进入个人 `feature` 分支，再合并到 `dev`，最后由吴宇轩将 `dev` 合并至 `main`。

## 3. 分支策略
- main
- dev
- feature-*

## 4. 命名约定
- **文档命名**：所有项目管理文档统一存放于 `docs/` 目录下，采用“序号-英文标识.md”格式（如 `01-project-charter.md`、`02-wbs.md` 等），确保文件顺序固定、类型清晰、便于检索。
- **分支命名**：严格遵循三层架构。`main` 为最终验收分支；`dev` 为日常集成分支；个人开发分支统一采用 `feature/<姓名拼音>-<职责>` 格式（如 `feature/liyixuan-wbs`、`feature/zhaoxinyu-config`），禁止使用模糊命名。
- **提交说明**：统一采用 `<type>(<scope>):<subject>` 格式。`type` 仅限实验允许的关键词（`feat`/`fix`/`docs`/`test`/`chore`/`merge` 等）；`scope` 标明修改模块；`subject` 使用中文简短描述（≤50字符，结尾不加标点），示例：`docs(wbs):补充三级WBS分解内容`、`fix(conflict):解决配置结论合并冲突`。

## 5. 合并规则
- **合并责任人与时机**：采用串行流水线合并机制。李奕轩负责合并吴宇轩基础分支至 `dev` → 赵新宇负责合并李奕轩分支及自身配置分支至 `dev` → 伍湘彬负责合并自身进度分支至 `dev` → 最终由项目经理吴宇轩将 `dev` 合并至 `main`。所有合并操作必须在个人分支推送完成、且已执行 `git pull origin dev` 同步最新代码后进行。
- **合并方式**：统一使用 `git merge --no-ff` 命令，强制生成合并节点提交，保留完整的分支拓扑历史。
- **冲突处理**：以 `docs/conflict.md` 的“当前结论”行作为预设冲突点。出现冲突时，严禁直接删除他人内容，需查看 `<<<<<<<`/`=======`/`>>>>>>>` 标记，对比双方修改意图后手工整合，综合保留核心要点。冲突解决后需单独提交（`fix(conflict):...`），并在总结报告中详细说明冲突产生原因与解决路径。

## 6. 版本留痕
- **分支隔离留痕**：通过 `main`（稳定版）← `dev`（集成版）← `feature/*`（个人开发版）的物理隔离，天然保存各阶段开发快照与演进过程。
- **提交记录核验**：强制要求本地配置 `git config user.name` 为本人拼音全称。实验结束后执行 `git shortlog -sne --all`，确保输出中完整包含 `wuyuxuan`、`liyixuan`、`zhaoxinyu`、`wuxiangbin` 四人的独立提交统计，实现贡献可追溯。
