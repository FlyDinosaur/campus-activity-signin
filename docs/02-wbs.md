# 02 WBS 文档

> 要求：WBS 必须围绕交付物（成果）分解，至少分解到三级。

## 1. 一级交付物
本组 WBS 以最终需要提交和检查的成果为主线，而不是以“讨论、开会、写代码”等过程动作作为主线。一级交付物划分如下：

1. 项目管理与过程文档集
2. 系统基础架构与运行环境
3. 活动发布与维护功能模块
4. 学生报名与审核功能模块
5. 现场签到与状态管理功能模块
6. 统计报表与系统配置功能模块

其中，项目管理成果用于说明项目背景、目标、角色和边界；范围计划成果用于明确三级 WBS；进度计划成果用于安排关键活动；配置管理成果用于保证 GitHub 协作过程可追溯；示例代码成果只作为项目结构支撑，不作为本实验重点。

## 3. 分解说明
本WBS分解严格遵循项目管理标准（PMBOK指南）的**“面向可交付成果（Deliverable-Oriented）”**原则，结合轻量级校园活动系统的业务特征与实验交付要求，采用**“业务闭环为主线、技术基座为支撑、交付物清单为边界”**的结构化拆解方法。具体分解依据如下：
### 一、 一级交付物划分依据
一级结构未采用传统的“需求-设计-开发-测试”阶段划分，而是以**“可独立验收的成果模块”**为核心，依据如下：
1. **业务闭环映射**：直接对应项目核心目标中的完整用户操作链路。`3.活动发布` → `4.报名审核` → `5.现场签到` → `6.统计导出`，形成端到端的功能主干，确保业务流程不中断。
2. **交付物范围强制覆盖**：实验明确要求提交“仓库结构、WBS、进度计划、配置管理、总结报告”。为便于追踪与验收，独立设立 `1.项目管理与过程文档集`，将过程资产作为一等交付物管理。
3. **技术基座隔离**：将数据库设计、权限控制、轻量鉴权等底层支撑独立为 `2.系统基础架构与运行环境`，避免业务代码与基础设施耦合，符合“流程清楚、记录完整”的系统设计初衷。
4. **100%规则与范围控制**：6个一级项完全覆盖“纳入范围”的全部功能，且严格剔除支付、深度对接、高并发优化、移动端App等“不纳入范围”内容，确保WBS边界清晰、无隐性膨胀。
### 二、 二级与三级分解依据
二三级结构遵循**“功能内聚、逻辑分层、粒度可控”**的拆解逻辑，依据如下：
1. **按业务流与数据流拆解**：沿“用户交互 → 业务处理 → 状态持久”路径展开。例如报名模块拆为 `4.1 在线报名处理`（前端提交+后端校验）与 `4.2 名单审核`（列表展示+批量操作+流水记录），确保每个二级包具备完整的输入输出闭环。
2. **按技术实现分层设计**：三级工作包按“表现层/组件 → 业务逻辑/规则 → 数据/配置”分层。如签到模块拆为 `5.1 二维码组件`（生成与展示）与 `5.2 身份核验组件`（匹配逻辑与状态机），前后端职责清晰，便于并行开发与单元测试。
3. **工作包颗粒度控制**：三级交付物严格控制在“可独立编码、可独立测试、可独立提交”的粒度（如 `1.1.2 分支管理规范`、`4.1.2 重复报名拦截代码`、`6.2.1 Excel导出工具类`）。符合学生团队敏捷迭代习惯，单个工作包预计耗时在8~40小时内，便于进度跟踪与责任到人。
4. **配置与运维轻量化预留**：将系统参数配置独立为 `6.2`，既满足“维护系统参数与基础配置”的角色要求，又以配置表+简单维护页面的形式实现，避免过度设计，契合“轻量级”定位。
